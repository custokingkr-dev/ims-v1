# Java 21 → 25 Pass Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move all 5 Java services from Java 21 → Java 25 (JVM runtime + compile target + CI + Docker images), staying on Spring Boot 4.0.7, with no application code changes.

**Architecture:** Two PRs. Task 1 (canary) bumps only CI `java-version` to 25 to prove the Mockito-agent / ByteBuddy / Testcontainers tooling runs on the JVM 25 runtime while the build target stays Java 21. Task 2 (full bump) sets `<java.version>25` in all 5 poms and updates all 5 Dockerfiles (build + runtime images) to Temurin 25. Then the controller deploys each service to prod incrementally and verifies. JDK 25.0.3 is installed locally, so validation runs both locally and in CI.

**Tech Stack:** Java 21→25 (Temurin), Maven (`mvnw.cmd`), Spring Boot 4.0.7, Mockito javaagent + ByteBuddy, Testcontainers, GitHub Actions CI, Docker (eclipse-temurin images), Cloud Run.

**Spec:** `docs/superpowers/specs/2026-07-04-java25-pass-design.md`

## Global Constraints

- **Target = Java 25** across all 5 Java services (billing, identity, operations, platform, school-core). The Node api-gateway and React frontend are NOT touched.
- **Stay on Spring Boot 4.0.7** — no framework change.
- **No application code changes.** The only permitted code touch is a dependency/tool **version** bump if the canary surfaces a Java-25 incompatibility (prefer BOM-managed; else an explicit Maven property). Do NOT weaken/skip/delete any test to pass on Java 25.
- **Behavior-preserving** — each service's existing suite is the oracle; test counts unchanged.
- **Local JDK 25** at `C:\Program Files\Java\jdk-25.0.3` (verified `25.0.3+9-LTS`). JDK 21 stays at `C:\Program Files\Java\jdk-21.0.11`. Select the JDK per command via `JAVA_HOME`.
- **CI `java-version`** is a single shared value in BOTH `.github/workflows/ci.yml` and `.github/workflows/whole-application-validation.yml` — bump both together.
- **Image tags:** runtime `eclipse-temurin:25-jre` (confirmed to exist). Build image `maven:<ver>-eclipse-temurin-25` — confirm the exact resolvable tag via `docker pull` before editing (current is `maven:3.9.9-eclipse-temurin-21`).
- **Incremental prod deploy** per service after the full-bump PR merges; mixed Java-21/25 fleet is safe; rollback = redeploy the prior Cloud Run revision.
- **Prod verify creds:** `e2e-superadmin@local.test` / `password`; `e2e-admin@local.test` / `password` (school 4). Gateway `https://custoking-api-gateway-xkv7oenbna-em.a.run.app`.

### Task 1: Canary — CI on JVM 25 (prove the tooling), poms/Dockerfiles stay 21

**Files:** `.github/workflows/ci.yml`, `.github/workflows/whole-application-validation.yml`. Branch: `phase2-java25-canary`.

**Interfaces:** Produces a CI configuration that runs every service's `mvn test` on JDK 25 while the compile target stays Java 21 — isolating the Mockito/ByteBuddy/Testcontainers-on-JVM-25 question. If a tool-version fix is needed, it lands here (a pom property) and carries into Task 2.

- [ ] **Step 1: Local spot-check — run a Mockito-agent + Testcontainers suite on JDK 25 first**

Pick two representative services: identity (Mockito unit tests + a Testcontainers integration test) and school-core (heavy multi-schema Testcontainers). Run each on **JDK 25** (Docker must be up for the integration tests):
```

.\mvnw.cmd -f services\identity-service\pom.xml clean test
.\mvnw.cmd -f services\school-core-service\pom.xml clean test
```
Expected: BUILD SUCCESS, each service's baseline test count, no `Unsupported class file major version` / ByteBuddy / Mockito-agent errors. **If a tooling error appears** (e.g. ByteBuddy rejecting class-file major 69), fix the tool **version** in that service's `pom.xml` — prefer overriding the SB-managed property (e.g. `<byte-buddy.version>…</byte-buddy.version>` or `<mockito.version>…</mockito.version>` to a Java-25-supporting release) — re-run until green, and note it so Task 2 replicates the same property to the other poms if needed. Record which services (if any) needed a fix.

- [ ] **Step 2: Bump CI `java-version` to 25 in both workflows**

In `.github/workflows/ci.yml` (line ~56) and `.github/workflows/whole-application-validation.yml` (line ~54), change `java-version: 21` → `java-version: 25`. Change nothing else. Do NOT touch poms or Dockerfiles in this task.

### Task 2: Full bump — poms `<java.version>25` + Dockerfiles Temurin 25

**Files:** the 5 service `pom.xml` (`<java.version>`) and 5 `Dockerfile` (build + runtime images). Branch: `phase2-java25-bump`.

**Interfaces:** Consumes Task 1 (CI already on JDK 25; any tool-version fixes already merged). Produces all 5 services compiling to Java 25 and packaged on Temurin 25 images.

- [ ] **Step 1: Confirm the exact maven Temurin-25 build-image tag**

```

```
If that tag resolves, use it. If it does NOT exist, try `docker pull maven:3.9-eclipse-temurin-25` and, failing that, the latest maven patch with a temurin-25 variant (e.g. `docker pull maven:3.9.11-eclipse-temurin-25`). Use the first tag that pulls successfully; record it as `<MVN25>` for Step 3. Also confirm the runtime image: `docker pull eclipse-temurin:25-jre` (expected to succeed).

### Task 3: Incremental prod deploy + verify (controller)

**Owner:** controller (no implementer). After Task 2 merges, deploy each of the 5 services on the Temurin-25 image and verify. Low-risk order: billing → operations → identity → platform → school-core.

- [ ] **Step 1: Deploy + verify each service (repeat per service)**

For `SVC` in that order:
```bash
gh workflow run deploy.yml --ref main -f environment=production -f deploy_services=<SVC>-service -f run_direct_smoke=false
# wait for the run to conclude success, then:
gcloud run services describe custoking-<SVC>-service --region asia-south2 --format='value(status.conditions[0].status, status.latestReadyRevisionName)'
```
Expect `success` + `Ready=True`. Confirm the service booted on Java 25 (boot log banner / no stack trace). Then verify an authenticated route through the gateway (non-5xx, real data):
```bash
GW="https://custoking-api-gateway-xkv7oenbna-em.a.run.app"
TOKEN=$(curl -s -X POST "$GW/api/v1/auth/login" -H "Content-Type: application/json" -d '{"email":"e2e-superadmin@local.test","password":"password"}' | python -c "import json,sys;print(json.load(sys.stdin)['accessToken'])")
# billing: /api/v1/sa/invoices ; operations: /api/v1/ff/requests ; identity: (login above) ; platform: /api/v1/audit-logs ; school-core: /api/v1/schools
echo "<route> -> $(curl -s -o /dev/null -w '%{http_code}' "$GW<route>" -H "Authorization: Bearer $TOKEN")"
```
Expected: 200/expected (not 5xx). Roll back that one service (redeploy prior revision) if it misbehaves, and stop to investigate.

- [ ] **Step 2: Final full-fleet verification**

```bash
GW="https://custoking-api-gateway-xkv7oenbna-em.a.run.app"
FE="https://custoking-frontend-xkv7oenbna-em.a.run.app"
echo "frontend: $(curl -s -o /dev/null -w '%{http_code}' $FE)"
TOKEN=$(curl -s -X POST "$GW/api/v1/auth/login" -H "Content-Type: application/json" -d '{"email":"e2e-superadmin@local.test","password":"password"}' | python -c "import json,sys;print(json.load(sys.stdin)['accessToken'])")
for p in "/api/v1/schools" "/api/v1/students?schoolId=4" "/api/v1/fees/bands?schoolId=4" "/api/v1/audit-logs" "/api/v1/sa/invoices"; do
  echo "$p -> $(curl -s -o /dev/null -w '%{http_code}' "$GW$p" -H "Authorization: Bearer $TOKEN")"
done
TOKEN2=$(curl -s -X POST "$GW/api/v1/auth/login" -H "Content-Type: application/json" -d '{"email":"e2e-admin@local.test","password":"password"}' | python -c "import json,sys;print(json.load(sys.stdin)['accessToken'])")
echo "admin /students -> $(curl -s -o /dev/null -w '%{http_code}' "$GW/api/v1/students" -H "Authorization: Bearer $TOKEN2")  /schools -> $(curl -s -o /dev/null -w '%{http_code}' "$GW/api/v1/schools" -H "Authorization: Bearer $TOKEN2")"
```
Expected: frontend 200; superadmin routes 200 with data; admin `/students` 200 (scoped), `/schools` 403 (RLS intact). This completes the Java 25 pass — all 5 services on Java 25 in prod.
