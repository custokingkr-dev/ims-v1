# Spring Boot 4.0 Rollout (4 services) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate billing, operations, platform, and school-core from Spring Boot 3.5.16 → 4.0.7 (staying on Java 21), one PR per service in 2 risk-ordered waves, deploying each to prod incrementally — reusing the recipe proven by the identity spike.

**Architecture:** Each service is its own branch → migrate → validate (full suite green + local boot) → PR → merge → prod deploy → verify. Implementer subagents do the per-service **code migration** and stop at a committed branch; the **controller** does PR/CI/merge/deploy/verify between tasks (prod deploys are controller-operated, like the identity spike's rollout). Wave 1 (billing, operations) is mechanical pom edits; Wave 2 (platform, school-core) adds Jackson-3 code migration and school-core's custom-Flyway discover-and-fix.

**Tech Stack:** Java 21, Maven (`mvnw.cmd`), Spring Boot 3.5.16 → 4.0.7, Jackson 2→3 (`com.fasterxml.jackson`→`tools.jackson`), Flyway (+ `spring-boot-flyway` module), Testcontainers 1.x→2.0.5, Docker Compose, GitHub Actions CI, Cloud Run.

**Spec:** `docs/superpowers/specs/2026-07-04-sb4-rollout-design.md` · **Spike recipe:** `docs/SB4-SPIKE-FINDINGS.md`

## Global Constraints

- **Stay on Java 21** (`<java.version>21</java.version>` unchanged) — SB 4.0.7 + Java 21 is supported. Java 25 is a separate later pass. Do NOT touch Dockerfiles or CI `java-version`.
- **Target = Spring Boot 4.0.7** exactly.
- **One service per branch/PR**, named `phase2-sb4-<service>` off `main`. Do not bundle services.
- **Behavior-preserving.** Flyway migrations are authoritative (never edit them); each service's existing test suite is the oracle (never weaken/delete tests). For Jackson-migrated files, **characterize** any genuinely-changed JSON shape with an assertion rather than deleting a test.
- **The recipe (per the fingerprint):** parent→4.0.7 (all) · `spring-boot-flyway` (all except school-core, where it's investigated) · Testcontainers artifactId rename (operations, platform, school-core — NOT billing) · `spring-boot-restclient` (platform only) · Jackson-3 code migration (platform 5 files, school-core 2 files). **No jjwt in any of the 4.**
- **Exact new-dependency coordinates** (BOM-managed, no `<version>`): `org.springframework.boot:spring-boot-flyway`, `org.springframework.boot:spring-boot-restclient`. **Testcontainers rename:** `org.testcontainers:postgresql`→`org.testcontainers:testcontainers-postgresql`, `org.testcontainers:junit-jupiter`→`org.testcontainers:testcontainers-junit-jupiter` (keep `<scope>test</scope>`).
- **Maven on Windows/PowerShell:** `$env:JAVA_HOME='C:\Program Files\Java\jdk-21.0.11'; $env:Path="$env:JAVA_HOME\bin;$env:Path"` then `.\mvnw.cmd -f services\<svc>\pom.xml <goal>`.
- **Incremental prod deploy** per service after merge; mixed SB4/SB3.5 fleet is safe. Rollback = redeploy the prior Cloud Run revision.
- **Prod verification creds** (existing test accounts): `e2e-superadmin@local.test` / `password` (superadmin), `e2e-admin@local.test` / `password` (school-admin, school 4). Gateway: `https://custoking-api-gateway-xkv7oenbna-em.a.run.app`.

## File Structure (per service)

- `services/<svc>/pom.xml` — the primary edit surface (parent, Flyway/RestClient module, Testcontainers rename).
- `services/<svc>/src/main/java/**` — Jackson-3 import/API renames (platform: 5 files; school-core: 2 files) and, for school-core, possibly `…Application.java` / `application.yml` for the Flyway auto-config exclude.
- `services/<svc>/src/test/java/**` — only characterization assertions for changed Jackson-3 JSON shapes.
- No Dockerfile / CI / migration changes.

```xml
        </dependency>
```
and rename the two Testcontainers artifacts (`postgresql`→`testcontainers-postgresql`, `junit-jupiter`→`testcontainers-junit-jupiter`, keep group + test scope).

- [ ] **Step 3: Migrate the 5 files from Jackson 2 → Jackson 3**

In each of the 5 files (and any importing test), rename imports and API from Jackson 2 to Jackson 3:
- `import com.fasterxml.jackson.databind.*` → `import tools.jackson.databind.*` (e.g. `ObjectMapper`, `JsonNode`, `ObjectReader/Writer`). **Exception:** annotations stay — `com.fasterxml.jackson.annotation.*` (e.g. `@JsonProperty`, `@JsonIgnore`) are UNCHANGED in Jackson 3.
- `import com.fasterxml.jackson.core.*` → `import tools.jackson.core.*` where used.
- Jackson-3 API deltas to handle as the compiler flags them: `ObjectMapper` construction (prefer `JsonMapper.builder().build()` / injected `ObjectMapper`), and checked exceptions — Jackson 3's `ObjectMapper` throws unchecked `JacksonException` (no more checked `JsonProcessingException`), so remove now-invalid `throws`/`try-catch` the compiler rejects. Make each change minimal and behavior-preserving.

- [ ] **Step 4: Compile, then loop until it compiles**

```
.\mvnw.cmd -f services\platform-service\pom.xml clean test-compile
```
Fix each `[ERROR]` with the Jackson-3 equivalent (Step 3 rules). Repeat until BUILD SUCCESS (compile).

- [ ] **Step 5: Full suite green + characterization pass**

```
.\mvnw.cmd -f services\platform-service\pom.xml "-U" clean test
```
Expected: BUILD SUCCESS, `Tests run: N` == baseline. Watch the notification/PubSub/reporting JSON tests specifically: Jackson 3 auto-registers all classpath modules (vs Jackson 2's opt-in), which can change (de)serialization. If a test fails because the emitted/parsed JSON shape legitimately changed, **characterize** the intended shape by fixing the assertion to the correct SB4 output (do not weaken the test) and note it. If the shape change would alter an over-the-wire contract (Pub/Sub payloads, notification bodies), STOP and report — that's a real behavior change to escalate, not silently accept.

```

docker compose logs --tail=40 platform-service
```
Expected: `Started PlatformServiceApplication` (or app main class), no stack trace, Flyway applied, no missing-bean errors (RestClient wired). Do NOT `docker compose down`.

```bash

```
