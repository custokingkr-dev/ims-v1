# Reporting Decoupling — SP1: school-core outbox engine + `tenant_school` reference dimension

**Date:** 2026-07-07
**Status:** Approved (design)
**Services:** `school-core-service` (new outbox engine + reference events), `platform-service`
(reporting inbox routing + new dimension read-models + read-swaps), `deploy/gcp` (Pub/Sub topic +
subscription).

**Program context:** This is **SP1 of 6** in the reporting event-projection decoupling program.
Reporting-service (merged into platform-service) still directly reads six other services' schemas.
The billing→reporting path is already event-decoupled; every other source emits **nothing**. SP1
builds the missing event-emission engine in school-core and uses it to give reporting local copies
of the school "dimension" data (schools, sections, academic years), so later SPs can project the
facts (fee/student/attendance/catalog) and join them against these local dims. Boundary rule
(`DEVELOPMENT_GUIDE.md`): cross-service access only via HTTP APIs or Pub/Sub event projection — SP1 removes two
direct cross-schema reads and lays the dimension groundwork.

---

## Context — what exists today

**Billing outbox (the proven pattern to mirror), `services/billing-service/.../outbox/`:**
`OutboxWriter` (append), `OutboxRelay` (`@Scheduled` poll unpublished → publish → mark
`published_at`), `DomainEventPublisher` iface + `PubSubDomainEventPublisher` (→ topic) +
`LoggingDomainEventPublisher` (local fallback), `OutboxPublisherConfiguration`, `EventEnvelope`
(record, `ims.event-envelope.v1`), plus `billing.outbox_events` table (`V3__outbox_events.sql`).
Known-good config lessons (from the billing rollout — see memory): the topic env var is
`BILLING_OUTBOX_PUBSUB_TOPIC_ID` (the `_ID` suffix matters — a mismatch silently disabled the real
publisher), the relay pod needs **min-instances=1** (a scaled-to-zero relay never publishes), and
the Pub/Sub **push subscription must use OIDC auth** to the private reporting endpoint.

**Reporting inbox (consumer), `services/platform-service/.../`:**
`api/internal/ReportingPubSubPushController` — `POST /api/v1/pubsub/reporting-events` (token
`REPORTING_PUBSUB_PUSH_TOKEN`, validates envelope, dedupes by `eventId`, writes
`reporting.reporting_event_inbox`). `application/ReportingEventInboxProcessor` (`@Scheduled`) drains
`RECEIVED` inbox rows, routes by `eventType` (today only `billing.invoice-upserted.v1`), and
projects into local read-models (`persistence/BillingInvoiceReadRepository` — idempotent
`ON CONFLICT` upsert; the model for our new dim repos).

**School-core mutation points to hook (`SchoolStructureReadRepository`):** `INSERT INTO
tenant_school.schools` (~line 185) + the school update path; `INSERT INTO
tenant_school.school_sections` (~line 332) + section update; academic-year create/activate. These
run inside `@Transactional` methods — the outbox append must be in the **same transaction**
(transactional outbox) so an event is emitted iff the domain change commits.

**Reporting reads SP1 swaps (`ReportingReadRepository`):**
- Active-year resolution: `SELECT id FROM tenant_school.academic_years WHERE active = true LIMIT 1`
  (lines ~412, ~493, ~581).
- Active-schools KPI: `SELECT count(*) FROM tenant_school.schools` (~line 731).
These are the only reads that are **purely** `tenant_school` (no join to fee/student facts). The
name-joins inside the fee-defaulter / low-attendance / class-photography queries stay cross-schema
until SP2–SP4 project those facts (they will then join the SP1 dims). *(Reporting's old
`tenant_school.school_timetable_entries` read was already removed by the timetable revamp.)*

## Decisions (locked during brainstorming)

1. **Event-projection** (not HTTP) — mirror the billing outbox pattern exactly.
2. **SP1 scope = the engine + the `tenant_school` reference dimension only** (schools, sections,
   academic years). Facts (fee/student/attendance/catalog/firefighting) are SP2–SP6.
3. **Backfill via a Flyway migration that seeds `outbox_events` from existing rows** — automatic,
   forward-only, no manual post-deploy step; the relay publishes them like any event.
4. **Eventual consistency is acceptable** for reporting KPIs (the relay poll interval lag).

## Data model

### school-core (`tenant_school` schema, new migration `V7…`)

**`tenant_school.outbox_events`** — mirror `billing.outbox_events`:
```sql
CREATE TABLE tenant_school.outbox_events (
    id             BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    event_key      VARCHAR(255) NOT NULL,      -- business idempotency key, e.g. "SchoolUpserted:12"
    event_type     VARCHAR(120) NOT NULL,      -- dotted versioned fact
    aggregate_type VARCHAR(64)  NOT NULL,
    aggregate_id   VARCHAR(255) NOT NULL,
    school_id      BIGINT,                      -- nullable (academic_years is global)
    occurred_at    TIMESTAMPTZ  NOT NULL DEFAULT now(),
    payload        JSONB        NOT NULL,
    published_at   TIMESTAMPTZ,
    attempts       INTEGER      NOT NULL DEFAULT 0,
    created_at     TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_ts_outbox_unpublished ON tenant_school.outbox_events (id) WHERE published_at IS NULL;
```
RLS is **not** applied to `outbox_events` (it is written by the same RLS-exempt/owner paths and
drained by the relay as a system component — mirror billing, whose outbox has no tenant policy;
the relay runs with the service identity, not a tenant GUC). Grant the runtime role
SELECT/INSERT/UPDATE + sequence USAGE (conditional `app_rt` grant, matching the timetable
migrations).

**Backfill migration** (same or a following `V…`): seed one event per existing row so history flows
through the pipeline:
```sql
INSERT INTO tenant_school.outbox_events (event_key, event_type, aggregate_type, aggregate_id, school_id, payload)
SELECT 'SchoolUpserted:'||id, 'school.upserted.v1', 'School', id::text, id,
       jsonb_build_object('id',id,'name',name,'shortCode',short_code,'city',city,'state',state,'active',active)
FROM tenant_school.schools;
-- analogous INSERT…SELECT for school_sections → 'school-section.upserted.v1' and
-- academic_years → 'academic-year.upserted.v1'.
```

### platform-service (`reporting` schema, new migration continuing its history)

Three dimension read-models, each idempotent-upsertable by aggregate id (mirror
`billing_invoice_read`; RLS consistent with the existing reporting read-models — reporting is a
platform/superadmin surface, so follow whatever `billing_invoice_read` does for RLS/grants):

- **`reporting.dim_school`** (`id BIGINT PK, name, short_code, city, state, active BOOLEAN, updated_at`)
- **`reporting.dim_section`** (`id VARCHAR PK, name, school_id BIGINT, class_id VARCHAR, class_name,
  active BOOLEAN, teacher_name, updated_at`) — carries denormalized `class_name` so a separate
  `dim_class` is unnecessary in SP1.
- **`reporting.dim_academic_year`** (`id VARCHAR PK, label, active BOOLEAN, updated_at`)

## Components

### school-core outbox engine (new `…/outbox/` package, mirroring billing)
- `EventEnvelope` (copy billing's record — `ims.event-envelope.v1` fields).
- `DomainEventPublisher` iface; `PubSubDomainEventPublisher` (publishes to topic from
  `SCHOOL_CORE_OUTBOX_PUBSUB_TOPIC_ID`, builds the canonical envelope + Pub/Sub attributes);
  `LoggingDomainEventPublisher` (local/dev fallback when the topic id is unset);
  `OutboxPublisherConfiguration` selecting the publisher; a startup check that logs which publisher
  is active (mirror `OutboxPublisherStartupCheck`, using the correct `…_TOPIC_ID` property name).
- `OutboxWriter` — `append(String eventType, String eventKey, String aggregateType, String
  aggregateId, Long schoolId, Map<String,Object> payload)` → `INSERT INTO tenant_school.outbox_events`.
  Called **inside the caller's `@Transactional`** so it commits atomically with the domain change.
- `OutboxRelay` — `@Scheduled` fixed-delay: `SELECT … WHERE published_at IS NULL ORDER BY id LIMIT
  N FOR UPDATE SKIP LOCKED`, wrap each in `EventEnvelope`, `publish(...)`, `UPDATE … SET
  published_at = now()`. At-least-once (reporting dedupes by `eventId`).

### school-core event emission (hook into `SchoolStructureReadRepository`)
Add `OutboxWriter.append(...)` in the same transaction as:
- **School create + update** → `school.upserted.v1`, key `SchoolUpserted:{id}`, aggregate `School`,
  payload `{id,name,shortCode,city,state,active}`.
- **Section create + update** → `school-section.upserted.v1`, key `SchoolSectionUpserted:{id}`,
  aggregate `SchoolSection`, payload `{id,name,schoolId,classId,className,active,teacherName}`.
- **Academic-year create/activate** → `academic-year.upserted.v1`, key `AcademicYearUpserted:{id}`,
  aggregate `AcademicYear`, payload `{id,label,active}` (school_id null). *(Find the year
  create/activate path; if years are seed-only today, still emit on the activate/toggle path and
  rely on the backfill for existing rows.)*

### reporting projection (platform-service)
- Extend `ReportingEventInboxProcessor` to route the three new `eventType`s to a new
  `DimensionProjectionRepository` with three idempotent upserts (`ON CONFLICT (id) DO UPDATE …`,
  guarded so an older event can't overwrite newer state if `occurred_at`/updated ordering matters —
  simplest: last-writer-wins by processing order, acceptable for dimension data).
- No change to `ReportingPubSubPushController` (it already accepts any envelope and routes by type).

### read-swaps (`ReportingReadRepository`)
- Active-schools KPI (~line 731): `count(*) FROM reporting.dim_school WHERE active = true`.
- Active-year resolution (~lines 412/493/581): `SELECT id FROM reporting.dim_academic_year WHERE
  active = true LIMIT 1`. (Replace the three inline `tenant_school.academic_years` reads with a
  shared private helper reading the dim.)
- Do **not** touch the fee/student/attendance name-join queries yet (SP2–SP4).

### infra (`deploy/gcp`)
- A new Pub/Sub **topic** for school-core's outbox (env `SCHOOL_CORE_OUTBOX_PUBSUB_TOPIC_ID`) and a
  **push subscription** delivering to reporting's `POST /api/v1/pubsub/reporting-events` with **OIDC
  auth** (mirror the billing topic/subscription assets). School-core service (the relay) needs
  **min-instances=1**. Add the env wiring in `cloudbuild.yaml` / Cloud Run config alongside the
  billing equivalents.

## Error handling & edge cases

- **Transactional outbox:** append in the same `@Transactional` as the mutation — no event on
  rollback, exactly-one on commit. The relay gives at-least-once delivery; reporting's inbox dedupe
  (`eventId`) makes projection idempotent.
- **Relay scaled to zero / topic id unset:** startup check logs the active publisher; in dev the
  `LoggingDomainEventPublisher` is used and nothing breaks. Document the min-instances=1 +
  `…_TOPIC_ID` requirements (repeat of the billing lessons) so the prod rollout doesn't silently
  no-op.
- **Backfill idempotency:** the seed migration runs once; if re-run defensively, dedupe by
  `event_key` (reporting also dedupes). Existing `dim_*` rows upsert.
- **Out-of-order events:** last-writer-wins on the dim (acceptable for reference data). If a delete
  of a school/section is later needed, add a `…deleted.v1` event in a follow-up — **SP1 handles
  upserts only** (schools/sections/years are not hard-deleted in normal flows).
- **RLS:** reporting dims follow the existing reporting read-model convention; the school-core
  outbox table is a system table (no tenant policy), drained by the relay.

## Testing (backend; platform + school-core Testcontainers + Mockito)

- **school-core:** `OutboxWriter.append` inserts a row in the caller's transaction (rolls back with
  it); the school/section/academic-year mutation paths emit the right `event_type`/`event_key`/
  payload; `OutboxRelay` publishes unpublished rows via a stub `DomainEventPublisher` and marks
  `published_at`; the backfill migration seeds one outbox row per existing school/section/year.
- **platform-service:** `ReportingEventInboxProcessor` routes each new event type into the correct
  dim upsert (idempotent — processing the same event twice yields one row); the swapped
  active-schools KPI + active-year reads return from the dims. (Testcontainers run RLS-exempt —
  assert projection/read correctness, not RLS filtering.)
- Keep `scripts/audit-service-authorization-boundaries.ps1` semantics intact (no new cross-schema
  read introduced; two removed).

## Build phasing (within SP1)

- **A — school-core outbox engine + emission + backfill** (migrations, `outbox/` package, hooks,
  tests). Independently shippable (events flow to the topic / logging publisher).
- **B — reporting dims + projection + read-swaps** (migration, processor routing, dim repos, swap
  the two reads, tests). Depends on A's event contract.
- **C — infra wiring** (`deploy/gcp` topic/subscription + Cloud Run env + min-instances). Can be
  prepared alongside; required before the prod rollout actually flows events (dev uses the logging
  publisher until the topic is wired).

## Out of scope (later SPs / follow-ups)

- Fee, student, attendance, catalog (SP2–SP5) and firefighting (SP6) fact projections — and the
  name-join queries that currently cross-read `tenant_school` alongside those facts (they swap to
  the SP1 dims once the facts are local).
- `…deleted.v1` events / tombstoning of dims (SP1 is upsert-only).
- A `dim_class` table (class_name is denormalized onto `dim_section`).
- Retiring the direct `tenant_school` grants from reporting's DB role (only safe once ALL
  tenant_school reads are gone — after the later SPs).
