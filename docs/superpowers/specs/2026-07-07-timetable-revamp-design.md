# Timetable Revamp — Design

**Date:** 2026-07-07
**Status:** Approved (design)
**Service:** `school-core-service` (owns `tenant_school` schema) + `frontend`. Read currently
lives in `platform-service`'s workspace bundle and is being moved to a dedicated school-core
endpoint.

---

## Context

The current timetable module is a placeholder:

- `frontend/src/pages/workspace/panels/TimetablePanel.tsx` is a flat, **append-only list** — no
  edit, no delete, no grid.
- Every field is **free-text**: `period_label`, `class_section` ("9-B"), `subject`, `teacher` —
  no dropdowns, no referential integrity, so typos and duplicates are unpreventable
  (`tenant_school.school_timetable_entries`, migration `V4__school_timetable.sql`).
- **No time model** (period is a label, no start/end), **no conflict detection**, **no RLS** on
  the table, and the read is buried in the `/api/v1/workspace` bundle with a hard `LIMIT 200`.
- **Missing master data**: no subjects table, no bell-schedule/periods table, no rooms.
  `school_classes` (shared catalog: `id`, `name`, `sort_order` — no `school_id`),
  `school_sections` (`id`, `name`, `teacher_name`, `active`, `school_class_id`, `school_id`),
  `staff_members` (`id`, `name`, …, `school_id`), and `academic_years` (`id`, `active`) exist.

This revamp turns it into a real, grid-based timetable with proper master data, edit/delete, times,
conflict warnings, and per-academic-year scoping with locking of past years.

## Decisions (locked during brainstorming)

1. **Grid + real CRUD**, reusing existing master data (sections, staff) plus new master data.
2. **Per-grade/stage bell schedules**: admin defines a few **named** schedules (e.g. "Primary",
   "Secondary"), each with its own ordered periods + times; **each class is mapped to one
   schedule**. A section's grid rows come from its class's schedule.
3. **Subjects master is per class, per academic year**, managed under the **Setup** nav group.
   Editable only for the **active** year; past years render **read-only (locked)**.
4. **The whole timetable is academic-year-scoped.** The active year is editable; past years are
   read-only archives (viewable, not changeable). Subjects and timetable entries both carry
   `academic_year_id`.
5. **Conflict = warn but allow.** A teacher double-booked in the same day+period elsewhere raises a
   non-blocking warning; the admin may still save. A section slot holds exactly one entry
   (setting a new subject/teacher **replaces in place**; clearing deletes).
6. **RLS everywhere.** All new tables and the reworked timetable table get row-level security
   consistent with the student/fee/attendance/catalog pattern.

"Active year" = the single `tenant_school.academic_years` row with `active = true`
(resolved the same way as elsewhere: `ORDER BY active DESC, id DESC LIMIT 1`). A year is editable
**iff it is the active year**; every other year is read-only.

## Data model

All tables live in `tenant_school`. New Flyway migrations continue that schema's history
(`services/school-core-service/src/main/resources/db/migration/tenant_school/`, next is `V5…`;
verify the highest existing version at implementation time and continue from there). Every table
below gets the standard RLS policy:

```sql
ALTER TABLE tenant_school.<t> ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON tenant_school.<t>
  USING      (school_id = nullif(current_setting('app.current_school_id', true), '')::bigint
              OR current_setting('app.bypass_rls', true) = 'on')
  WITH CHECK (school_id = nullif(current_setting('app.current_school_id', true), '')::bigint
              OR current_setting('app.bypass_rls', true) = 'on');
```

### `school_bell_schedules` (new)
Named bell schedules per school.
- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `school_id BIGINT NOT NULL REFERENCES tenant_school.schools(id)`
- `name VARCHAR(120) NOT NULL`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- `UNIQUE (school_id, name)`

### `school_bell_periods` (new)
Ordered periods of a schedule; these are the **grid rows**.
- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `school_id BIGINT NOT NULL REFERENCES tenant_school.schools(id)`
- `schedule_id BIGINT NOT NULL REFERENCES tenant_school.school_bell_schedules(id) ON DELETE CASCADE`
- `sort_order INTEGER NOT NULL`
- `label VARCHAR(64) NOT NULL` (e.g. "P1", "Break")
- `start_time TIME NOT NULL`
- `end_time TIME NOT NULL`
- `is_break BOOLEAN NOT NULL DEFAULT false` (break rows show in the grid but hold no subject/teacher)
- `UNIQUE (schedule_id, sort_order)`
- CHECK `end_time > start_time`

### `school_class_bell_map` (new)
Maps a school's class to one bell schedule.
- `school_id BIGINT NOT NULL REFERENCES tenant_school.schools(id)`
- `class_id VARCHAR(255) NOT NULL REFERENCES tenant_school.school_classes(id)`
- `schedule_id BIGINT NOT NULL REFERENCES tenant_school.school_bell_schedules(id) ON DELETE CASCADE`
- `PRIMARY KEY (school_id, class_id)`

A class with no mapping has no schedule → its sections' grids prompt the admin to assign one in
Setup before a timetable can be built.

### `school_class_subjects` (new)
Per-class, per-year subjects master.
- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `school_id BIGINT NOT NULL REFERENCES tenant_school.schools(id)`
- `class_id VARCHAR(255) NOT NULL REFERENCES tenant_school.school_classes(id)`
- `academic_year_id VARCHAR(255) NOT NULL REFERENCES tenant_school.academic_years(id)`
- `subject_name VARCHAR(120) NOT NULL`
- `sort_order INTEGER NOT NULL DEFAULT 0`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- `UNIQUE (school_id, class_id, academic_year_id, subject_name)`

### `school_timetable_entries` (reworked)
The existing table is replaced (its free-text columns are dropped; the handful of legacy free-text
rows in dev are not migrated). New columns:
- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `school_id BIGINT NOT NULL REFERENCES tenant_school.schools(id)`
- `academic_year_id VARCHAR(255) NOT NULL REFERENCES tenant_school.academic_years(id)`
- `section_id VARCHAR(255) NOT NULL REFERENCES tenant_school.school_sections(id) ON DELETE CASCADE`
- `day_name VARCHAR(16) NOT NULL` (Mon…Sat; validated app-side against the allowed set)
- `bell_period_id BIGINT NOT NULL REFERENCES tenant_school.school_bell_periods(id) ON DELETE CASCADE`
- `subject_name VARCHAR(120) NOT NULL` (denormalized from the class's subjects master at write
  time — keeps history if the master later changes; the write validates it is a current member)
- `teacher_id BIGINT REFERENCES tenant_school.staff_members(id) ON DELETE SET NULL` (nullable —
  a slot may hold a subject with no teacher assigned yet)
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- `UNIQUE (school_id, academic_year_id, section_id, day_name, bell_period_id)` — one entry per
  slot (enables replace-in-place upsert; a section never holds two subjects in one slot)

Migration note: because the reworked columns are incompatible with the old free-text ones, the
migration **drops and recreates** `school_timetable_entries` (safe — dev only, minimal data) and
adds RLS (which the old table lacked). This is an intentional forward-only rework recorded in the
migration comment.

**Known simplification:** bell schedules/periods are school-level config, not year-scoped. Editing
a schedule's periods reshapes the grid rows for all years, including past-year archives (the
archived entries persist and are shown against the current period rows). Year-scoping the bell
schedule is out of scope for this revamp.

## Setup area (admin — under the **Setup** nav group)

Two new panels reachable from the admin `Setup` nav group (`config.ts`).

### Bell Schedules panel
- List named schedules; create/rename/delete a schedule.
- Within a schedule: add/edit/reorder/delete periods (label, start, end, break flag). Reorder via
  up/down (updates `sort_order`).
- **Class → schedule assignment**: a matrix/list of the school's classes (derived from the classes
  that have sections in this school) each with a schedule dropdown; saving upserts
  `school_class_bell_map`.

### Subjects Master panel
- Pick a **class** (and the year defaults to the active year; a year selector lets you view past
  years read-only).
- For the active year: add/remove/reorder subjects for that class (`school_class_subjects`).
- For a past year: the list renders **read-only** with a "Locked — <year label> has ended" notice;
  add/remove controls are hidden/disabled. The server also enforces this (see API).

## Timetable grid (main UX — admin edits, teacher views read-only)

Replaces `TimetablePanel.tsx`.

- **Controls**: class-section dropdown (from `school_sections`), and a year selector (defaults to
  active). Non-active years are read-only.
- **Grid**: rows = the section's class's bell-schedule periods (label + `start–end` time down the
  left; break rows are visually distinct and non-editable). Columns = Mon–Sat.
- **Cell**: shows `Subject` + `teacher name` when set; empty otherwise. Clicking a cell (active
  year, admin) opens an inline editor with a **Subject** dropdown (from the class's year subjects)
  and a **Teacher** dropdown (from `staff_members`). Save upserts the slot; a "Clear" action
  deletes it.
- **Conflict warning** (non-blocking): before/at save, if the chosen `teacher_id` is already
  assigned in the same `academic_year_id + day_name + bell_period_id` for a **different** section,
  show an inline warning naming the clash (e.g. "RK already teaches 8-A · Mon P2") with a
  **Save anyway** affirmation. Saving proceeds.
- **Empty states**: if the section's class has no bell-schedule mapping → prompt to set it up in
  Setup. If the class has no subjects for the year → prompt to add subjects in Setup.
- **Teacher role**: same grid, **read-only** (no cell editing; no year is editable). Teachers reach
  it from their `Classroom` nav.

## API (school-core-service)

New dedicated endpoints (compat `/api/v1/**`, routed via the gateway to school-core/"tenant";
add gateway routes for the new prefixes). All are internal-token gated (`X-Tenant-School-Token` /
existing tenant-school token) **and** end-user permission gated, and resolve the school via
`TenantScope`. Reads use `resolveSchoolId` (own school; superadmin may pass a schoolId). Writes are
admin-only and always operate on the caller's own school.

**Bell schedules**
- `GET  /api/v1/timetable/bell-schedules` → schedules + their periods for the school.
- `POST /api/v1/timetable/bell-schedules` (name) / `PUT …/{id}` (rename) / `DELETE …/{id}`.
- `POST /api/v1/timetable/bell-schedules/{id}/periods` (label,start,end,is_break,sort_order) /
  `PUT …/periods/{periodId}` / `DELETE …/periods/{periodId}`.
- `GET  /api/v1/timetable/class-schedules` → `[{classId, className, scheduleId}]` for the school.
- `PUT  /api/v1/timetable/class-schedules/{classId}` (scheduleId) → upsert `school_class_bell_map`.

**Subjects master**
- `GET  /api/v1/timetable/class-subjects?classId=&yearId=` → subjects + `{ editable: <bool> }`
  (`editable` = year is active).
- `POST /api/v1/timetable/class-subjects` (classId, subjectName) — **rejected 409/400 if the
  target year is not the active year** (server-enforced lock). `DELETE …/{id}` — same guard.

**Timetable**
- `GET  /api/v1/timetable?sectionId=&yearId=` → `{ editable, days, periods:[{id,label,start,end,
  isBreak}], entries:[{day, periodId, subjectName, teacherId, teacherName}] }`. `editable` = active
  year AND caller is admin. Resolves the section's class → schedule → periods.
- `PUT  /api/v1/timetable/entry` (sectionId, day, periodId, subjectName, teacherId) → upsert the
  slot. Guards: year must be active (else 409 locked); `subjectName` must be a current member of
  the class's year subjects (else 400); period must belong to the section's class's schedule and
  not be a break (else 400). Returns the saved entry **plus** a `conflict` field describing any
  teacher double-booking (non-blocking).
- `DELETE /api/v1/timetable/entry?sectionId=&day=&periodId=` → delete the slot (active year only).

The legacy `POST /api/v1/workspace/timetable` write and the `timetable` key in the
`/api/v1/workspace` bundle (`platform-service` `ReportingReadRepository.timetable`) are **removed**;
`TimetablePanel` stops reading from the bundle and calls the dedicated endpoints.

## Roles & permissions

- Wire the existing (currently-unenforced) permission codes: **`timetable:manage`** gates all
  writes (bell schedules, class-schedule map, subjects, timetable entries) and **`timetable:read`**
  gates reads. Grant `timetable:manage` + `timetable:read` to ADMIN/SCHOOL_ADMIN;
  `timetable:read` to TEACHER. Superadmin bypasses.
- Nav: the two Setup panels are admin-only (Setup group). The timetable grid stays in admin
  Academics and teacher Classroom. Gate the panels/nav on the permission codes.

## Error handling & edge cases

- **Year lock**: enforced server-side on every subjects/timetable write (not just hidden in UI).
  Non-active year write → 409 with a clear "…year has ended and is read-only" message; the FE shows
  it and keeps the grid read-only.
- **Conflict**: computed on timetable read (annotate cells) and returned on save; never blocks.
- **Break periods**: cannot hold entries (400 if attempted).
- **Deletes/cascades**: deleting a schedule cascades its periods and (via `bell_period_id`
  cascade) the timetable entries in those periods; deleting a section cascades its entries;
  deleting a staff member sets `teacher_id` NULL (slot keeps its subject, teacher shows "—").
- **Subject removed from master while in use**: allowed (denormalized `subject_name` on entries
  keeps history); the master edit does not rewrite existing entries.
- **Section with no schedule / class with no subjects**: friendly empty-state prompts (above);
  writes that require them return 400 with guidance.

## Testing

Backend (school-core-service, Mockito + the existing Testcontainers pattern):
- Year-lock guard rejects writes to a non-active year (subjects + timetable).
- Timetable upsert is replace-in-place (unique slot); break-period and non-member-subject writes
  rejected.
- Teacher double-booking is reported (not blocked) and correctly identifies the clashing section.
- Tenant scoping: a non-superadmin cannot read/write another school's timetable; RLS present on
  the new tables (note: Testcontainers run RLS-exempt — assert the app-level `TenantScope` guards;
  RLS is defense-in-depth).
Frontend: build + a light render test of the grid (renders periods×days, shows a set entry).
Confirm at plan handoff whether to include the FE render test or match the earlier no-FE-test
revamps.

## Build phasing

- **Phase A — master data + Setup**: the four new tables + reworked timetable table (migrations,
  RLS), backend CRUD for bell schedules / class-schedule map / subjects (with year lock), and the
  two Setup panels. Independently shippable and testable.
- **Phase B — timetable grid**: the timetable read/upsert/delete API (conflict annotation), the new
  grid panel (admin edit + teacher read-only), removal of the legacy bundle read/write. Depends on
  A.

## Out of scope (later)

- Rooms/venues; teacher–subject qualification links; auto-generation/optimization of timetables.
- Year-scoped bell schedules (see Known simplification).
- Printing/PDF export of the timetable (can be a follow-up like the attendance reports).
- Substitutions / daily overrides / holidays.
- Per-section (as opposed to per-class) subject lists or schedules.
